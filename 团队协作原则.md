# 团队协作原则

> robot-servo-sim · 电气传动课程设计  
> 本文档是对 CONTRIBUTING.md 技术规范的补充，侧重**理念与决策逻辑**。

---

## 一、为什么要用 GitHub 管理 AI 对话？

AI 输出的结果**不可靠**——它可能编造型号、算错参数、遗漏约束。  
如果直接把 AI 的结论复制进报告或 Simulink 模型，一旦出错，没有人知道错误从哪来、是谁引入的、为什么当时采纳了它。

GitHub 仓库解决的核心问题是**可追溯性**：

- 每一个结论，都能追溯到对应的 Prompt 和原始输出
- 每一个修正，都记录在校验笔记里
- 每一次进展，其他人不用微信问，直接 `git pull` 看

## 二、三个"不能省"的原则

### 原则 1：Prompt 写成文件，先 commit 再发给 AI

不能直接在聊天框里打 Prompt 然后截图存起来。  
**必须**先把 Prompt 写成 `.md` 文件 commit 进仓库，再把内容发给 AI。

原因：
- 其他人能看到你准备问什么，提前给意见
- 形成"问题提出"的时间戳，报告写 AI 协作留痕时有据可查
- 即使 AI 给出很差的回答，你也有"我问了什么"的记录

### 原则 2：原始输出一字不改地存进仓库

不要"帮 AI 优化一下"再存进 `02_responses/`。  
**必须**原汁原味地存，无论回答多么混乱或啰嗦。

原因：
- 校验笔记里"错误的部分"需要和原文对照
- 如果改了原文，校验记录失去意义，也无法复现错误

### 原则 3：校验笔记是含金量最高的文件，不能敷衍

校验笔记不是"总结"，它是**人工核查过程的完整记录**，必须包含：
- 可信度标签
- 正确的部分（说清楚为什么正确）
- 错误的部分（原文 → 修正，逐条列出）
- 最终采用的结论
- 给下游成员的交接信息

敷衍的校验笔记（比如只写"基本正确，可用"）等于没做校验，对团队没有价值。

## 三、两种 commit 的本质区别

这是很多人会困惑的地方，一句话说清楚：

> **过程记录**（日记）→ 直接 push main，随时可见，不需要任何人批准  
> **关键结论**（里程碑）→ 开 PR，等至少 1 人确认后才能合并

判断是否需要开 PR 只有一个问题：  
**"有没有其他人要把我的这个输出当作已知条件，开始做下一步工作？"**

- S2 的一个 v1 Prompt 草稿 → 直接 push main
- S2 最终确定的电机型号和 Rs/Ls/Ke 参数 → 开 PR，因为 S3 和 S4 都要用

```
过程中的每一步 commit（随时 push main）
    ↓ ↓ ↓ ↓ ↓
    [P commit] [C 对话] [R commit] [追问 commit] ...
                                        ↓
                               得出最终结论
                                        ↓
                              开 PR → review → merge
                                        ↓
                          结论进入 main，下游可以依赖
```

## 四、"接力"规则：如何引用他人的工作成果

S4 需要 S2 的电机参数时，**不发微信问**，而是：

```bash
git pull
# 打开 03_checks/S2_mech/001_..._校验.md
# 读"最终采用的结论"和"下一步"章节
```

然后在自己的 Prompt 里明确引用来源：

```markdown
## 已确认的电机参数（来源：03_checks/S2_mech/001_..._校验.md）
- 电机类型：BLDC
- 额定功率：XXX W
- Rs = XX Ω，Ls = XX mH，Ke = XX V·s/rad
```

**为什么这样做？**
- 引用有来源，方便审查
- 如果 S2 后来修正了参数，S4 能在 commit history 里看到变更
- 报告里"各模块如何协作"的叙述有具体文件作为证据

## 五、参数引用铁律

仓库里所有数值参数的**唯一真相源**是 `00_frozen/params.md`。

- 写 Prompt 时：打开 `params.md`，复制粘贴，**不得凭记忆手打**
- 写 `init_params.m` 时：也从 `params.md` 查，不要从别处抄
- 如果发现 `params.md` 有误：找 S1，由 S1 修改并开 PR 通知全员

**一旦有人用了不同来源的参数，整套仿真结果就会出现无法解释的数值不一致。**

## 六、.slx 文件的特殊地位

Simulink 的 `.slx` 文件是**二进制文件**，Git 无法显示它的差异，也无法自动合并。两个人同时编辑同一个 `.slx` 会导致严重的版本冲突，只能手动重做。

所以 `.slx` 适用独立的互斥锁规则（见 README.md 的"模型编辑锁"表格）：

1. 要编辑 `.slx` 前，先在 README.md 填锁
2. 编辑完立即 push 并清锁
3. 下一个人拉取最新版本再编辑

这不是繁文缛节，是防止"两人各改了半天，最后只能保留一个人的版本"的必要措施。

## 七、进度看板的使用规范

README.md 顶部的进度看板**每人每天更新自己那行**，不超过 5 分钟。

更新规则：
- 状态变化时立刻更新（不要等到第二天）
- "最新文件"填 `Sx_xxx/NNN`（目录名 + 流水号），不用写全路径
- "阻塞点"要写清楚：等谁做什么，而不是"等上游"这种模糊描述

看板的目的是**消除信息不对称**，让每个人不用问就能知道全组状态。

## 八、模型选择建议（按任务类型）

| 任务类型 | 推荐模型 | 原因 |
|---------|---------|------|
| 公式推导、参数计算 | GPT-5 / Claude Opus | 数学推理能力强 |
| 大量文本整理、报告撰写 | Claude Sonnet | 长文本生成质量高 |
| 代码生成（MATLAB/Python） | GPT-5 | 代码格式规范 |
| 图表分析、波形解读 | Gemini | 多模态能力强 |
| 快速草稿、思路发散 | 任意模型 | 速度优先 |

> ⚠ 无论使用哪个模型，校验步骤都不能省略。模型越强大，幻觉越难被识别。

## 九、发生冲突时怎么办

### Git 文本文件冲突（.md 文件）
```bash
git pull
# 如果有冲突，VS Code 会高亮显示
# 手动选择保留哪个版本，或合并两者
git add .
git commit -m "解决合并冲突"
git push
```

### .slx 文件冲突（两人同时编辑了）
1. 两人都不要再改
2. 找 S1 仲裁，决定以哪个版本为准
3. 另一个人的修改手动重新做一遍（教训：下次先填锁）

### 参数不一致（不同文件用了不同值）
1. 以 `00_frozen/params.md` 为准，其他文件全部更正
2. 更正后通知所有人 `git pull`

## 十、报告写作时的 AI 协作留痕

最终报告需要说明"AI 参与了哪些环节、人工如何校验修正"。  
仓库的 PCR 记录是最好的证据，写报告时直接引用：

- `01_prompts/` → "我们向 AI 提出了以下问题……"
- `02_responses/` → "AI 的原始回答如下……"
- `03_checks/` → "经人工核查，发现以下问题并进行了修正……"

**不需要额外整理，仓库本身就是 AI 协作过程的完整留痕。**
